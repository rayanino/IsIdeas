# Idea Specification Template

Use this template for every serious idea.

---

# [Idea Name]

## 1. Thesis
One sentence: what this system is and why it matters.

## 2. User Problem
What exact recurring study bottleneck does this solve?

## 3. Why This Matters for Islamic Study
Why is this strategically important in this domain specifically?

## 4. Strategic Value
How does this increase the user's long-term scholarly capacity?

## 5. Scope Boundary
What is in scope, and what is explicitly out of scope?

## 6. Primary Users
Who is this designed for?
Examples:
- early student,
- advanced student,
- researcher,
- teacher-supported student.

## 7. Inputs
What data does the system consume?
Examples:
- taxonomy topics,
- source metadata,
- excerpts,
- user questions,
- memorization history,
- teacher notes.

## 8. Outputs
What does the system produce?
Examples:
- plans,
- queues,
- flags,
- comparisons,
- summaries,
- graphs,
- alerts.

## 9. Core Workflow
Describe the end-to-end user flow.

## 10. Core Entities / Data Model
List the main entities and their relationships.

## 11. System Behavior
Describe the main logic.
Examples:
- ranking,
- scheduling,
- retrieval,
- matching,
- conflict detection,
- scoring.

## 12. Source-Backed vs Inferred Data
What must be directly traceable, and what may be inferred or computed?

## 13. Correctness Constraints
What must the system never do incorrectly or silently?

## 14. Human Review Requirements
Where must a human approve, interpret, or override?

## 15. Failure Modes
How can this system mislead, overreach, or waste time?

## 16. MVP
What is the smallest version that proves the core value?

## 17. V2 / V3 Expansion
What later capabilities become possible if the MVP succeeds?

## 18. Evaluation Criteria
How will we know whether the system is actually valuable?

## 19. Dependencies
What other systems, data contracts, or repo primitives does this rely on?

## 20. Open Questions
What still needs careful design work?

## 21. Claude Code Implementation Brief
Provide a concise build brief that an implementation agent can act on.

---

## Optional Appendix
Use appendices for:
- sample schemas,
- candidate UI surfaces,
- algorithm sketches,
- edge cases,
- example scenarios.
