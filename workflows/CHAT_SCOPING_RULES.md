# Chat Scoping Rules

These rules help keep idea work clean across future chats.

## 1. Prefer one serious target per chat
A good default is one chat for:
- one idea,
- one repo-level architecture concern,
- or one cross-cutting comparison question.

## 2. Start a new chat when the target changes materially
Start a fresh chat when moving from:
- one idea to a different idea,
- idea work to repo architecture work,
- or exploratory discussion to implementation hardening.

## 3. Use long chats deliberately, not lazily
Continue a chat only while it still helps coherence.
Do not keep one chat alive merely out of convenience.

## 4. New chat triggers
Strong candidates for a new chat include:
- context becoming crowded,
- too many open threads inside one discussion,
- the need for a fresh deep dive,
- or fear that important nuance will be diluted.

## 5. Repo updates should survive the chat
A chat is valuable only if its conclusions make it back into repository artifacts.
The repository, not the chat history, should hold the durable state.

## 6. One chat should usually end with one clear repo consequence
Examples:
- dossier strengthened,
- idea parked,
- stage promoted,
- decision recorded,
- open question refined,
- or architectural rule added.
