# Idea Generation Protocol

Use this protocol when generating new software ideas for the workshop.

The goal is not to maximize novelty.
The goal is to discover ideas that are worthy of serious attention.

## Core Rule

Start from bottlenecks, not features.

A weak generation session begins with:
- "what app could be cool?"

A strong generation session begins with:
- "what repeatedly wastes scholarly effort?"
- "what repeatedly weakens understanding, retention, or correctness?"
- "what seems hard today because the right software layer does not exist?"

## Preferred Order

### 1. Name the bottleneck
Describe the recurring pain, confusion, friction, or waste.

### 2. Ask whether software is actually the right tool
Not every important problem is a software problem.

### 3. Generate multiple candidate responses to the same bottleneck
Do not marry the first idea.

### 4. Compare the candidates brutally
Ask:
- which one creates the most leverage?
- which one is safest?
- which one best compounds with the rest of the portfolio?
- which one is most likely to be extraordinary rather than merely decent?

### 5. Preserve only what survives challenge
Move survivors into `capture/` or, if warranted, into the serious registry.

## Anti-Patterns

Avoid idea-generation sessions that mainly produce:
- generic productivity tools
- UI fantasies
- novelty with no durable leverage
- ideas that depend on magical intelligence rather than good system design
- ideas that sound scholarly but do not actually improve study

## Rule Of Restraint

An unrecorded weak idea is not a loss.
A preserved weak idea that later steals serious attention is a cost.
